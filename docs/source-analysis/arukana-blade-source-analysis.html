3.2 データ二重保存の仕組み

重要： このプロジェクトはゲームデータを2ヶ所に保存しています。
① localStorage（即時・オフラインでも動作）と ② PostgreSQL（1.5秒デバウンス自動保存）の二重保存です。 
ゲームプレイ（操作）
        ↓
Zustandストア（メモリ上の状態変更）
        ↓
        ├──→ [即時] localStorage に自動保存
        │          Zustand の persist 機能で自動管理
        │
        └──→ [1.5秒デバウンス] syncService.ts
                    ↓
             collectGameState() → 全ストアをJSONに
                    ↓
             POST /api/player/saveAll
                    ↓
             Player.gameStateJson = 全状態JSON
             + 正規化カラム（gold, diamond, etc.）同期

ページ再読み込み時:
  GET /api/auth/me → gameStateJson取得
  → hydrateFromGameState() → 全ストア復元

第7章　API一覧

7.1 認証API

POST /api/auth/register — 新規ユーザー登録

項目内容
認証必要不要
リクエスト{ email: string, password: string }
バリデーションemail: 正規表現形式チェック / password: 最低8文字
処理bcrypt.hash(pw,12) → User作成 → Player作成 → JWT → Cookie
成功レスポンス{ user, player }
エラー400: バリデーション失敗 / 409: メール重複 / 500: サーバーエラー
呼び出し元src/features/auth/RegisterPage.tsx

POST /api/auth/login — ログイン

項目内容
リクエスト{ email: string, password: string }
処理DB検索 → bcrypt.compare → ログイン日数更新 → JWT → Cookie
成功200: { user, player }
エラー401: 認証失敗
呼び出し元src/features/login/LoginPage.tsx

GET /api/auth/me — 認証確認・ゲームデータ取得

項目内容
認証必要必要（Cookie: arcana_session）
処理JWT検証 → Player取得（gameStateJson含む）
成功200: { user, player }（player.gameStateJson = 全ゲーム状態）
呼び出し元authStore.checkAuth()（App.tsx起動時）

POST /api/auth/logout — ログアウト

項目内容
処理arcana_session Cookieを無効化（expiredに設定）
成功200: { ok: true }
呼び出し元authStore.logout()

7.2 プレイヤーAPI

POST /api/player/save — プロフィール保存

項目内容
認証必要必要
リクエスト{ playerName?, tutorialCompleted?, title?, bio?, favoriteUnitId? }
バリデーションplayerName: 最大12文字 / bio: 最大100文字
処理Player.update()（指定フィールドのみ更新）

POST /api/player/currency — 通貨・スタミナ保存

項目内容
認証必要必要
リクエスト{ gold?, diamond?, exp?, playerRank?, stamina?, maxStamina? }
バリデーション各値 ≥ 0
呼び出し元バトル勝利後、チュートリアル完了後

POST /api/player/saveAll — 全ゲーム状態一括保存

項目内容
認証必要必要
リクエスト{ state: { ...全ゲーム状態 } }
処理gameStateJson = state（全体） + 正規化カラム同期
呼び出し元syncService.ts の自動保存（1.5秒デバウンス）

7.3 ゲームAPI

POST /api/summon/save — ガチャ結果保存

項目内容
認証必要必要
リクエスト{ poolId, units: [{masterId, rarity, resultType}], diamondSpent }
resultTypenew=新規追加 / crystal=被り（覚醒結晶）
処理（トランザクション）OwnedUnit.createMany(new only) → SummonHistory.createMany → diamond減算
呼び出し元SummonPage.tsx, TutorialGachaScreen.tsx

POST /api/units/sync — ユニット全件同期

項目内容
認証必要必要
リクエスト{ units: [{masterId, level, exp, awakenRank, awakeningCount, currentRarity, isLocked}] }
処理（トランザクション）OwnedUnit.deleteMany（全削除）→ OwnedUnit.createMany（全再作成）
呼び出し元TutorialGachaScreen.tsx（チュートリアル完了後）

9.1 ユーザー登録フロー

9.2 ガチャ（召喚）フロー

画面:  SummonPage で「10連」ボタン押下
          ↓
フロント: playerStore.spendDiamond(150) ← ダイヤ消費
          ↓
         ガチャ計算（クライアント側）:
         ・レアリティ抽選（SSR 4%, SR 15%, R 40%, N 41%）
         ・排出ユニットを決定
          ↓
         unitStore.processSummonResults(masterIds)
         ├── 新規ユニット → OwnedUnit追加（type:'new'）
         └── 被りユニット → 覚醒結晶付与（type:'crystal'）
          ↓
API:     POST /api/summon/save
         { poolId, units:[{masterId, rarity, resultType}], diamondSpent:150 }
          ↓
DB:      トランザクション:
         OwnedUnit.createMany（new のみ）
         SummonHistory.createMany（全件記録）
         Player.diamond -= 150
          ↓
画面:    ガチャ演出 → 結果表示

9.3 バトルフロー

クエストマップでステージ選択
          ↓
スタミナ消費（playerStore.spendStamina）
          ↓
BattlePage初期化:
  自軍: partyStore.getActiveParty() のユニット（最大5体）+ フレンド
  敵軍: enemies.ts のマスターデータ
          ↓
【プレイヤーターン】

src/utils/battleEngine.ts — バトルエンジン

ゲームのダメージ計算・スキル効果処理を担う。純粋関数（引数だけで結果が決まる関数）で実装されている。

// 属性相性テーブル（火→風は1.5倍など）
const ELEMENT_ADVANTAGE: Record> = {
  fire:  { wind: 1.5, water: 0.75 },
  water: { fire: 1.5, wind: 0.75 },
  wind:  { earth: 1.5, fire: 0.75 },
  earth: { water: 1.5, wind: 0.75 },
  light: { dark: 1.5 },
  dark:  { light: 1.5 },
};

// ダメージ計算
export function calcDamage(atk, def, power, atkElem, defElem, atkBuffs, defBuffs) {
  // 1. バフ/デバフを乗算
  const buffedAtk = atk * (1 + atkBuffs.reduce((a, b) => a + b, 0));
  // 2. 属性倍率
  const elementMult = ELEMENT_ADVANTAGE[atkElem]?.[defElem] ?? 1.0;
  // 3. 基本ダメージ（攻撃力×スキル倍率 - 防御力×0.5）
  const baseDmg = buffedAtk * power - def * 0.5;
  // 4. 乱数（±10%のブレ）
  return Math.floor(baseDmg * elementMult * (0.9 + Math.random() * 0.2));
}

14.1 認証の全体像

このプロジェクトは JWT（JSON Web Token）+ HTTP Only Cookie による認証を採用しています。

14.2 パスワードの安全な保管（bcrypt）

// ❌ 絶対にやってはいけない（平文保存）
DB: { passwordHash: "password123" }

// ✅ このプロジェクトの実装（bcryptハッシュ）
const hash = await bcrypt.hash("password123", 12);
DB: { passwordHash: "$2b$12$xxxxxxxxxxxxxxxxxxxxxxxx..." }

// ログイン時の照合（元のパスワードには戻せない）
const isValid = await bcrypt.compare("password123", storedHash);
// → true（一致） or false（不一致）
なぜbcryptを使うのか：
通常のハッシュ（MD5・SHA1等）は計算が速いため、大量のパスワードを総当たりで試される危険があります。 bcryptは意図的に遅い計算をするため、不正アクセスのコストが高くなります。 cost: 12 は計算の重さ（2^12回の反復）を表します。

14.3 JWTの仕組み

// 1. ログイン成功時にJWTを生成
const token = jwt.sign(

  process.env.JWT_SECRET,    // 署名用シークレットキー

);
// → "eyJhbGciOiJIUzI1NiJ9.eyJ1c2VySWQiOiJjbHh4eDEyMyJ9.xxxx"

// 2. 次のリクエスト時に検証
